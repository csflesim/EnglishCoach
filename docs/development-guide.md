# 開發與貢獻指南 (Development Guide)

歡迎參與 **English Coach** 的開發！本指南詳細說明本專案的程式碼規範、Git 分支流程與測試準則。

---

## 🌿 Git 分支與開發流程

本專案採用 **GitHub Flow** 工作流：

1. **分支命名規範**：
   - 功能開發：`feat/feature-name`
   - 錯誤修正：`fix/bug-name`
   - 文件更新：`docs/doc-name`
   - 效能優化：`perf/optimization-name`
2. **提交訊息規範 (Commit Messages)**：
   我們遵循 [Conventional Commits](https://www.conventionalcommits.org/) 規範：
   - `feat: 新增 AI 對話存檔功能`
   - `fix: 修正 TTS 語音按鈕點擊無反應問題`
   - `docs: 更新快速上手步驟`
3. **發送 Pull Request (PR)**：
   - 開發完成後，向 `main` 分支發起 PR。
   - 必須通過所有自動化測試與 ESLint 檢查。
   - 需獲得至少一名開發者的 Approve 始可合併。

## 🎨 程式碼風格與規範

為確保程式碼一致性，請遵守以下工具之配置：

- **ESLint**: 靜態程式碼檢查，執行 `npm run lint` 檢查。
- **Prettier**: 程式碼格式化。建議在編輯器中設定「存檔時自動格式化 (Format on Save)」。
- **TypeScript**: 必須定義明確的型別，避免濫用 `any`。

## 🧪 測試指南 (Testing)

我們使用 **Jest / Vitest** 進行單元與整合測試，並使用 **Playwright** 進行端到端 (E2E) 測試。

### 執行單元測試：
```bash
npm run test
```

### 執行 E2E 測試：
```bash
npm run test:e2e
```

- **撰寫測試要求**：
  - 新增核心 Utility 函式或 API 路由時，應同步撰寫單元測試。
  - 關鍵業務邏輯覆蓋率（Code Coverage）需保持在 **80%** 以上。

## 🧪 核心測試案例 (Core Test Cases)

### 1. 操練流程測試 (Drill Flow Tests)
- **不打斷**：學習者一旦開口(VAD 偵測到或手動「我開始說了」)，系統**不得**在說話途中切換或顯示正解；必須等到偵測靜音(說完)後才進入 reveal。
- **反應速度量測**：反應秒數 = cue 播完到開口的時間，並正確分級(`<1.5s` Excellent／`1.5–3s` Good／`3–6s` Learning／`>6s` Weak)。3 秒只是參考線，**不得**作為強制給答案的倒數。
- **四式分開 + 兩層選擇**：一個 session 只跑**單一模式**;不得四式合併連跑。流程為 選句型 → 選模式 → 第二層 → 操練。情境反應目前關閉。
- **替換(選句框) + 單字庫來源**：替換選定句框後，**變數由 `vocabulary` 依句框 `category` 抓取**(非寫死);每句框約 20 發,字數不足才少;進句框先示範。
- **轉換(選句框 → 選操作)**：轉換須能**由使用者選擇要變換哪個句框**(非寫死 substitution[0]),再選操作;以該句框的整批句子逐句字串轉換(`enFrom→enTo`、`zhFrom→zhTo`，疑問再 `.→?`/`。→嗎？`);**不得**只有過去式。
- **擴展(抽 5 輪替)**：每次抽 5 條擴展鏈，下次換另外 5 條;每條 = 基句 + 逐層加長(最多 5 層);進基句先示範。
- **麥克風備援**：麥克風不可用時須自動切換為手動兩鍵模式，邏輯一致。
- **TTS 不被切斷**：朗讀以 `onend` 驅動推進,**不得**用固定計時導致長句(中/英)被切斷。
- **Echo Loop（可選）**：開啟時 reveal 須跑 `Target → [覆述停頓] → Native → Target → [覆述停頓]`，每次 Target 後留覆述時間;母語取自 `nativeZh`(非即時翻譯)；關閉時只顯示一次正解。

### 2. 弱點引擎測試 (Weakness Engine Tests)
- **答錯/慢反應入庫**：答錯或反應過慢的項目須寫入 `weakness_items`，`mistake_count` 遞增、`status` 標為 `weak`。
- **反覆出現直到掌握**：弱點項目須在同一 session 稍後與隔天再次出現，連續快速答對才可晉升 `mastered`；未掌握前不得移除。
- **弱點頁可操練**：弱點頁的句型項目點選後須能直接開始針對該句型的操練。

### 3. 學習推進測試 (Progression Tests)
- **完成記錄持久化**：完成一個模式後須記錄;重整頁面(reload)進度不消失(MVP 用 `localStorage`)。
- **單元掌握**：一個單元的所有可用模式都完成 → 標記掌握;首頁顯示 ✓ 與「X/30」。
- **下一步推薦**：首頁 Next Up 須指向**第一個尚未掌握**的已建好單元(非寫死)。

### 4. 內容來源測試 (Content Source Tests)
- **不靠 AI 生成課程**：句型與學習路徑一律來自 `patterns` 與 FSI 30 單元地圖(受控內容)，**嚴禁**用 AI 隨機生成課程或教文法;母語錨點 `nativeZh` 亦為預存,非即時翻譯。
- **AI 僅限兩用途**：AI 只用於 STT 與回應評估;評估須回傳 `accuracy`/`feedback`/`weakness` 標籤並正確觸發弱點引擎。

### 5. 學習內容語言測試 (Content Language Tests)
- **學習內容全英文**：句型與單字等學習內容保持全英文;操作引導可用中文,但**不得**翻譯學習內容本身。

### 6. AI 評分與自由發揮測試 (AI & Free Practice Tests)
- **手動開關**：AI 評分預設關閉;🤖 開啟後才錄音/上傳/評分。
- **金鑰伺服器端**：`OPENAI_API_KEY` 只在 `/api/*` 使用,**不得**出現在前端 bundle。
- **優雅退回**：無金鑰或 API 失敗時,`lib/ai.ts` 回 `null`,操練須退回免費模式(只量反應速度、播正解),**不得**崩潰或卡住。
- **評分輸出**：評分須回傳 `correct/accuracy/grammar/fluency/feedback/weakness` 並寫入弱點引擎。
- **自由發揮**：每回合後接 3 題、或可直接進入;**不給 cue**,由學習者自造句;每題 AI 評分+記錄;未開 AI 時此模式不提供。
- **回合制成本**：用 Whisper + gpt-4o-mini + 瀏覽器 TTS;**嚴禁**用 Realtime API 做操練。

### 7. AI 補詞與單字複習測試 (Vocab AI & Review Tests)
- **AI 只豐富變數**：`/api/suggest-words` 只建議單字(英+中),須**存入詞庫(source=ai)可審可刪**;**不得**用 AI 生成句型結構或即時隨機(結構/中文模板維持受控)。
- **每回合變化**:詞庫足量時,每回合替換的 20 字須從該分類池**隨機抽一批**,不固定。
- **使用次數**:每練一個單字,`used_count` +1;`last_seen` 更新。
- **回合勾選不熟**:回合結束可勾選本回合用過的單字 → 標 `Weak` → 進弱點 + 單字複習專區。
- **單字複習專區**:列出不熟/弱單字(卡片),複習表現更新熟悉度。
